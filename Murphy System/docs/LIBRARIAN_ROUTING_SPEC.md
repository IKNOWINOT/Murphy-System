## Overview

This document specifies the Librarian-driven task routing system that replaces the hardcoded `IntegrationBus` chains in Murphy System. The Librarian acts as the central inference layer between any incoming task and the 600+ capabilities registered in the module registry.

The core principle: **no hardcoded routes**. Every task goes through the Librarian, which consults the live capability map and returns a ranked list of solution paths. Gates validate each path before execution.

## Schedules, Conditions, and Domain Gate Generation

The routing pipeline above handles task arrival. This section describes how tasks *originate* and how domain rules are encoded into gate specifications that the Librarian uses to pre-filter capability paths.

### Task Origins: Schedulers

Three scheduler classes produce tasks that feed into `TaskRouter`:

#### `AutomationScheduler` — Project-based cron scheduling

Manages recurring automation tasks declared as `ProjectSchedule` objects. Each schedule has:
- `project_id` — logical name for the automation
- `task_type` — capability name (routed through the Librarian)
- `cron_expression` — standard cron syntax (`"0 6 * * *"` = daily at 06:00)
- `priority` — `critical | high | medium | low`
- `parameters` — passed to the capability at execution time

When a cron fires, `AutomationScheduler` creates a `ScheduledExecution`, submits it to the priority queue, and dispatches it through the normal Librarian → Gate → Wingman pipeline.

#### `AutonomousScheduler` — Dependency-aware task queue

Manages one-time tasks with explicit dependencies, deadlines, and retry policies. A `Task` may declare `dependencies: ["task-id-A", "task-id-B"]` — the scheduler will not start it until both dependencies have completed successfully. Supports automatic retry (`max_retries`) and deadline enforcement.

#### `GovernanceScheduler` — Authority-enforced scheduling

Wraps any task submission with three governance rules that cannot be bypassed:
1. **Authority Precedence** — a task from a higher authority band pre-empts a lower one
2. **Resource Containment** — tasks cannot start if they would exceed declared resource budgets
3. **Dependency Resolution** — declared dependencies must be satisfied before execution begins

A scheduling request can return `SCHEDULE`, `BLOCK`, `TERMINATE`, `ESCALATE`, or `REFUSE`. `REFUSE` is a valid terminal outcome — the scheduler does not throw an error, it returns a structured refusal with a reason code.

#### `AlertRulesEngine` — Metric-triggered task firing

The rules engine evaluates `AlertRule` definitions against a stream of observed metric values. When a rule's condition is satisfied, it fires an alert and optionally triggers a task submission:

```python
AlertRule(
    rule_id    = "scale-out-on-cpu",
    name       = "Scale out when CPU > 85%",
    severity   = AlertSeverity.CRITICAL,
    metric     = "cpu_utilization",
    comparator = Comparator.GT,
    threshold  = 85,
    cooldown_seconds = 300,
)
```

`Comparator` supports: `gt`, `lt`, `gte`, `lte`, `eq`. The `cooldown_seconds` field prevents duplicate alerts within the window.

### Conditions: The Atomic Unit of Decision Logic

A `GateCondition` is a single predicate: `parameter operator expected_value`. Conditions appear throughout the system:

- Inside `DomainGate.conditions[]` — what must be true for the gate to apply
- Inside `AlertRule` — what metric state triggers the rule
- Inside `GovernanceScheduler` checks — what authority/resource state allows scheduling
- Inside `TriggerCondition` objects (autonomous agent behavior triggers) — what agent state fires an action

```python
GateCondition(
    condition_id   = "conf-001",
    condition_type = "threshold",
    parameter      = "confidence_score",
    operator       = ">=",
    expected_value = 0.85,
    description    = "Execution confidence must meet minimum threshold",
)
```

A `GateAction` defines what happens when the condition result is known:

```python
GateAction(
    action_id   = "act-001",
    action_type = "block",      # proceed | retry | escalate | block | log | notify
    trigger     = "fail",       # fires when the condition evaluates to fail
    target      = "execution",
    parameters  = { "reason": "confidence below threshold" },
)
```

### Domain Generative Gates: Rules → Specification → Librarian Filter

This is the key mechanism that connects your domain's rules to the Librarian's routing decisions.

**Step 1: You declare domain requirements**

```python
requirements = {
    "domain":                   "healthcare",
    "complexity":               "complex",
    "regulatory_requirements":  ["hipaa", "soc2"],
    "security_focus":           True,
    "performance_requirements": {"max_latency_ms": 200, "max_error_rate": 0.001},
}
```

**Step 2: `DomainGateGenerator` consults `LibrarianKnowledgeBase`**

`LibrarianKnowledgeBase` holds structured domain knowledge organized by domain and category:

```
healthcare:
  best_practices:
    - minimum_necessary_access: "Only access PHI required for the task"
    - audit_every_access:       "All PHI access must be logged with identity"
    - data_at_rest_encrypted:   "PHI must be encrypted at rest (AES-256)"
  regulatory_standards:
    - hipaa: "HIPAA Privacy and Security Rules"
    - hitech: "HITECH breach notification requirements"
  architectural_requirements:
    - role_based_access:        "RBAC with minimum necessary permissions"
    - break_glass_procedure:    "Emergency access with post-hoc audit"
```

`LibrarianKnowledgeBase.get_gate_templates("healthcare")` converts each knowledge entry into a gate template. `DomainGateGenerator` then instantiates `DomainGate` objects from those templates, filling in the `GateCondition` list and wiring the correct validator function from the function registry:

```
knowledge entry: hipaa
    → DomainGate(
        name              = "hipaa_compliance_gate",
        gate_type         = GateType.COMPLIANCE,
        severity          = GateSeverity.CRITICAL,
        conditions        = [
            GateCondition(parameter="compliance_domain", operator="==", expected_value="healthcare"),
            GateCondition(parameter="phi_present",        operator="==", expected_value=True),
        ],
        wired_function    = "validate_compliance_hipaa",
        knowledge_references = ["healthcare.regulatory_standards.hipaa"],
        risk_reduction    = 0.9,
      )
```

The full gate spec for a `healthcare/complex/hipaa+soc2` domain might contain 12–18 gates. Each gate has a type, severity, conditions, wired validator function, and a `knowledge_references` list that links back into the Librarian's knowledge base so the Librarian can explain *why* a path was filtered.

**Step 3: Gate spec registers with `GovernanceKernel` and feeds back into the Librarian**

```
DomainGateGenerator.generate_gates_for_system(requirements)
    → List[DomainGate]
    → GovernanceKernel.register_domain_spec(domain, gates)
    → SystemLibrarian.update_domain_gate_spec(domain, gates)
```

From this point, `SystemLibrarian.find_capabilities(task)` operates in two phases:

1. **Pre-filter** — for each candidate capability, check whether its declared domain tags and compliance annotations are compatible with every `CRITICAL` and `HIGH` gate in the spec. Capabilities that cannot possibly satisfy a blocking gate are removed from the candidate list before scoring. This prevents the Librarian from ever suggesting an execution path that will deterministically fail at the gate.

2. **Score and rank** — score the surviving capabilities against the task using semantic matching, keyword overlap, domain tag alignment, and `FeedbackIntegrator` historical weights. Return top-N as `CapabilityMatch` objects.

**Step 4: The Librarian's response explains gate compatibility**

```json
{
  "query": "process patient intake form",
  "domain": "healthcare",
  "active_gate_spec": "healthcare_v1",
  "matches": [
    {
      "capability_id": "patient_intake_pipeline",
      "score": 0.96,
      "gate_compatibility": {
        "hipaa_compliance_gate": "compatible",
        "phi_encryption_gate":   "compatible",
        "audit_logging_gate":    "compatible"
      },
      "match_reasons": ["domain:healthcare", "keyword:intake", "keyword:patient"]
    },
    {
      "capability_id": "generic_form_processor",
      "score": 0.61,
      "gate_compatibility": {
        "hipaa_compliance_gate": "incompatible — no HIPAA annotation"
      },
      "filtered": true,
      "filter_reason": "CRITICAL gate 'hipaa_compliance_gate' cannot be satisfied"
    }
  ]
}
```

The filtered capability (`generic_form_processor`) is included in the response for observability — operators can see *what was excluded and why* — but it is never promoted to a `SolutionPath`.

## Business Context Generative Gates

> **This section describes a separate gate layer from the compliance/security/technical gates documented above. Those gates are static policy. These gates are dynamically generated from live business ratios, are specific to each customer, and are learned over time by shadow agents.**

The compliance gates ask: *"Is this action safe and legal?"*  
The business context gates ask: *"Is this the right action for this business right now?"*

Both layers are enforced. Neither replaces the other.

---

### The Core Idea

Every customer has a live business state: pipeline fill rate, conversion rate, capacity utilization, revenue gap, lead velocity. These numbers have ratios between them. Those ratios answer operational questions:

- No inbound leads → pipeline will empty in N weeks → marketing must fire now
- Conversion below benchmark → revenue goal requires more effort than capacity allows → hiring or subcontracting gate
- At 90% capacity → taking new work causes delivery failures → business development must pause, delivery automation must accelerate

These are not rules a human writes once. They are **functions agents generate from the customer's current business state and goals**, then test, then promote to production. Once solidified, they become the live gate policy for that customer. Shadow agents observe what actually moves the metrics and refine the thresholds over time.

---

### The Four Function Types

Every business context gate is composed of exactly four generated functions:

```
1. SENSOR     — read one metric from business state → scalar value
2. GO / NO-GO — evaluate the scalar against a threshold → binary decision
3. ACT / DON'T ACT — trigger an automation if GO, suppress it if NO-GO
4. NOTIFY     — alert the operator with the reason when the gate fires
```

**Example: pipeline fill rate gate for a `professional_services` customer**

```python
# 1. SENSOR — reads the ratio from live business state
def sensor_pipeline_fill_rate(state: BusinessStateSnapshot) -> float:
    """Current pipeline value as a fraction of the 90-day target."""
    return state.active_pipeline_value / state.pipeline_target_value

# 2. GO / NO-GO — evaluates the ratio against the customer's threshold
def gate_pipeline_below_target(state: BusinessStateSnapshot) -> GateDecision:
    """Pipeline is more than 40% below target → marketing required."""
    if sensor_pipeline_fill_rate(state) < state.thresholds["pipeline_fill_rate_min"]:
        return GateDecision.GO
    return GateDecision.NO_GO

# 3. ACT / DON'T ACT — routes to the right capability if gate opens
def act_fill_pipeline(state: BusinessStateSnapshot, router: TaskRouter) -> ActionDecision:
    if gate_pipeline_below_target(state) == GateDecision.GO:
        router.route({
            "task":        "fill_pipeline",
            "domain":      state.industry,
            "channels":    state.preferred_channels,
            "effort_units": _effort_required(state),
        })
        return ActionDecision.ACT
    return ActionDecision.DONT_ACT

# 4. NOTIFY — tells the operator why the gate fired
def notify_pipeline_warning(state: BusinessStateSnapshot, notifier: Notifier) -> None:
    rate = sensor_pipeline_fill_rate(state)
    gap  = state.pipeline_target_value - state.active_pipeline_value
    notifier.send(
        severity  = "warning",
        message   = (
            f"Pipeline is {rate:.0%} of target. "
            f"${gap:,.0f} gap. "
            f"Estimated {_effort_required(state):.0f} effort-units needed "
            f"to close by {state.revenue_period_end}."
        ),
        action_taken = "Queued: content_pipeline, sales_automation",
    )
```

The ratio math is explicit and visible:

```python
def _effort_required(state: BusinessStateSnapshot) -> float:
    """
    How much pipeline-building effort is needed to close the revenue gap?

    effort = revenue_gap / (avg_deal_size × conversion_rate)

    If effort > available_capacity:
        → hiring gate or subcontractor gate fires
    If effort ≤ available_capacity:
        → pipeline fill automation fires
    """
    revenue_gap       = state.revenue_target - state.projected_revenue
    deals_needed      = revenue_gap / state.avg_deal_size
    leads_needed      = deals_needed / state.conversion_rate
    return leads_needed / state.lead_to_opportunity_rate
```

### Industry-Specific Sensor Sets

Different industries read different metrics. The agent generating the gate functions starts from the customer's industry profile (sourced from `agentic_onboarding_engine.BUSINESS_DEMOGRAPHICS`) and selects the relevant sensor set:

| Industry | Key Sensors | Attention Metrics |
|---|---|---|
| `professional_services` | pipeline fill rate, billable utilization, conversion rate, proposal win rate | revenue gap, capacity headroom, pipeline velocity |
| `software_saas` | MRR, churn rate, LTV/CAC ratio, trial conversion, expansion revenue | NRR, payback period, net new MRR |
| `manufacturing` | OEE (overall equipment effectiveness), yield rate, defect rate, order backlog | throughput gap, inventory turns, delivery lead time |
| `construction` | bid-to-win ratio, job gross margin, change order rate, schedule variance | backlog months, subcontractor availability, cash flow runway |
| `retail` | inventory turnover, sell-through rate, foot traffic, basket size | stockout rate, shrink rate, margin per sq ft |
| `healthcare` | appointment utilization, no-show rate, billing cycle time, payer mix | denial rate, days in A/R, provider productivity |
| `logistics` | on-time delivery rate, fleet utilization, cost per mile, damage rate | idle time, route efficiency, driver turnover |

The same **four function types** apply to every industry. What changes is which metrics the sensors read and what thresholds trigger the gates.

### The State Machine: Generated → Solidified

A business context gate passes through six lifecycle states managed by `GateLifecycleManager`:

```
PROPOSED
    │  Agent generates the 4-function set from current business state.
    │  Functions exist as code artifacts, not yet active.
    ▼
GENERATED
    │  Artifact is stored. Conditions and thresholds are visible to
    │  the operator. Not yet evaluated against live data.
    ▼
TESTED
    │  Gate runs in shadow mode: sensor fires, go/no-go evaluated,
    │  but act/don't-act does NOT trigger real automation.
    │  Notify fires with prefix "[SHADOW]" so operator can review.
    │  Gate accumulates N shadow observations (default: 5 cycles).
    ▼
ACTIVE
    │  Gate is live. All four functions run against real business state.
    │  Act/don't-act routes real automations. Notify sends real alerts.
    │  Shadow agents observe outcomes (did the automation move the metric?).
    ▼
SOLIDIFIED
    │  Gate has accumulated enough validated observations (configurable:
    │  default 20 cycles with >60% metric-movement confirmation) that its
    │  thresholds are considered calibrated for this customer.
    │  Shadow agents continue observing but threshold updates require
    │  HITL approval before applying.
    ▼
RETIRED
       Gate's conditions no longer apply (e.g., business grew past the
       phase where pipeline fill was the constraint — now capacity is
       the constraint). GateLifecycleManager retires it and proposes
       replacement gates for the new business state.
```

### Shadow Agents: Learning Which Gates Actually Work

Shadow agents are the feedback mechanism that makes business context gates customer-specific over time.

Three shadow agent types participate in this layer:

**`ShadowAgent` (from `shadow_agent_integration.py`)** — mapped to a primary operator role as an org-chart peer. Observes the operator's decisions (which alerts they acted on, which automations they approved, which gates they bypassed) and records them to the agent's `_learning_data` store. This is the "apprentice watching the human" layer.

**`TradingPatternLearner` analog — `BusinessPatternLearner`** — runs the business context gates in parallel on paper (no real automations fire). After each observation window (default: one week), compares the paper outcome to the real outcome. If the paper run would have produced a better result (e.g., fired marketing earlier, at a lower pipeline threshold), records a `PatternProposal`:

```
PatternProposal {
  gate_id:              "pipeline_fill_rate_gate_v3",
  customer_id:          "acme-consulting",
  observed_threshold:   0.60,     # what the gate used
  optimal_threshold:    0.52,     # what would have produced better outcome
  confidence:           0.74,
  observation_count:    12,
  metric_delta:         +0.18,    # pipeline fill rate improved by 18pp after firing
  status:               PENDING,  # awaits operator review
}
```

**`OrgShadowLearner` (from `org_compiler/shadow_learning.py`)** — observes role-level task patterns (which tasks the operator repeatedly handles the same way) and proposes automation templates for HITL review. These become new `act_*` functions.

All proposals are sandbox-only artifacts. None modify production gate thresholds without explicit operator approval (`promote_pattern()` equivalent). Once promoted:

```
GateLifecycleManager.update_threshold(gate_id, new_threshold, source="shadow_promotion")
    → records in activation_log
    → notifies GovernanceKernel to refresh domain gate spec
    → notifies SystemLibrarian to reload domain gate spec
```

### How the Librarian Sees Business Context Gates

When `SystemLibrarian.find_capabilities(task, domain=customer_domain)` runs, it consults both gate layers:

```
Layer 1 — Technical/Compliance gates (static policy)
    Is this action safe? Legal? Authorized?
    → filters out structurally incompatible capabilities

Layer 2 — Business Context gates (dynamic, customer-specific)
    Is this the right action for this customer's business state right now?
    → ACTIVE and SOLIDIFIED gates contribute a business_fit_score per capability
    → capabilities whose act_* function would NOT fire for current business state
      are ranked lower (not filtered out — the state could change between now and execution)
    → capabilities whose act_* function WOULD fire are ranked higher

Combined ranking:
    final_score = librarian_semantic_score
                × feedback_historical_weight
                × business_fit_score          ← NEW: from business context gates
```

**Example routing for a `professional_services` customer with low pipeline:**

```
Task: "generate new client outreach"

Business state:
  pipeline_fill_rate:  0.38   (threshold: 0.60)  → pipeline gate: GO
  capacity_utilization: 0.55  (threshold: 0.80)  → capacity gate: NO-GO (not at capacity)
  conversion_rate:      0.24  (benchmark: 0.28)  → conversion gate: borderline

Librarian capability ranking:
  1. content_pipeline           score: 0.91  business_fit: 1.00  (pipeline GO, capacity OK)
  2. sales_automation           score: 0.87  business_fit: 0.95  (pipeline GO)
  3. social_media_scheduler     score: 0.79  business_fit: 0.90
  4. proposal_quality_analyzer  score: 0.71  business_fit: 0.88  (conversion borderline)
  5. subcontractor_network      score: 0.44  business_fit: 0.12  (capacity NO-GO — ranked down)
     "subcontractor_network not relevant: capacity_utilization 0.55 < 0.80 threshold"
```

The operator sees a ranked list that reflects the actual business situation. The `business_fit_score` explanation is surfaced in the HITL panel so the operator can see the reasoning — not just the number.

### `TaskRouter`

```python
from dataclasses import dataclass, field
from typing import Any, Dict, List, Optional
from enum import Enum


class RouteStatus(Enum):
    APPROVED   = "approved"
    HITL       = "hitl_required"
    BLOCKED    = "blocked"
    NO_PATH    = "no_viable_path"


@dataclass
class RoutingResult:
    task_id:        str
    status:         RouteStatus
    solution_path:  Optional["SolutionPath"]
    alternatives:   List["SolutionPath"]
    gate_results:   Dict[str, str]          # gate_name -> "pass"|"fail"|"hitl"
    confidence:     float                   # 0.0–1.0


class TaskRouter:
    """
    Librarian-first task router.

    Usage::
```

## Agent Domain Locking and the Activation Pipeline

> **Everything described in this section already exists in the codebase. This section documents how the existing modules wire together to implement domain locking and the generated → tested → active → graduated lifecycle.**

---

### The Problem It Solves

A plumbing company's agents must never generate content, automations, or gate functions about aerospace. A consulting firm's agents must never fire manufacturing-floor automations. The domain lock is what makes each customer's Murphy deployment specific to their business and incapable of drifting outside it.

---

### Step 1 — Description → Form Fill → Constraints (`InferenceDomainGateEngine.infer_via_llm`)

`src/inference_gate_engine.py` — `InferenceDomainGateEngine.infer_via_llm()`

Keyword classification is a fallback. The primary path uses the LLM to infer meaning from the customer's business description and map it directly into the `RosettaFormSchema`. The filled form is the transformer: its fields ARE the business constraints that drive everything downstream.

```
"We run a plumbing and drain cleaning business in Austin TX"
        │
        ▼
InferenceDomainGateEngine.infer_via_llm(description, llm_backend)
        │
        ├── Build RosettaFormSchema — all fields empty
        │     form.fields: [industry, business_type, company_size,
        │                   primary_goal, key_challenges, existing_tools, ...]
        │
        ├── Construct extraction prompt from form field definitions
        │     "Extract structured context. Return JSON with:
        │      industry, business_type, company_size, primary_goal,
        │      key_challenges, ..."
        │
        ├── SafeLLMWrapper.safe_generate(prompt, context)
        │     → _check_relevance() Jaccard gate on output
        │     → returns: { "industry": "trade_services",
        │                  "business_type": "plumbing contractor",
        │                  "company_size": "small",
        │                  "primary_goal": "grow service area in Austin",
        │                  "key_challenges": "scheduling, invoice collection",
        │                  ... }
        │
        ├── Feed each answer into form as SensorReading
        │     SensorType.LLM_INFERENCE + FillConfidence.LLM_GENERATED
        │     form.submit_answer("industry", "trade_services")
        │     form.submit_answer("business_type", "plumbing contractor")
        │     ...
        │     LLM_GENERATED fields flagged: ActionFormField.needs_gating = True
        │     → must pass confidence gate before accepted as constraint
        │
        ├── MAGNIFY — map_org_positions("trade_services", description)
        │     → field_technician, dispatcher, estimator, office_manager, ...
        │     generate_inferred_gates(description, "trade_services", positions)
        │     → safety gate, quality gate, authorization gate, budget gate, ...
        │     gates derived from the FILLED FORM FIELDS, not from a bucket label
        │
        └── SOLIDIFY — InferenceResult (locked ground truth)
                → inferred_industry: "trade_services"
                → org_positions: [field_technician, dispatcher, estimator, ...]
                → inferred_gates: [safety_gate, quality_gate, ...]
                → form_schema: fully filled — each field is a constraint
```

**The form fields are the constraints.** Industry is not a bucket — it's one of many filled fields. The LLM's semantic understanding of "plumbing contractor in Austin" fills `business_type`, `primary_goal`, and `key_challenges` with specific content that keyword matching could never produce. Those fields directly determine which gates are generated, which org positions are created, and which automation pipeline flows are built.

**Fallback path:** When no LLM backend is available, `infer_via_llm()` falls back to `infer()` which uses `INDUSTRY_KEYWORDS` for classification. `INDUSTRY_KEYWORDS` covers all industries in `BUSINESS_DEMOGRAPHICS` — it is the safety net, not the primary mechanism.

### Step 1b — Form Fields → Gates → AgentCallToAction Pipeline

`src/inference_gate_engine.py` — `AgentActionBuilder`

Once the `InferenceResult` is produced, `AgentActionBuilder.build_actions_from_inference()` converts every org position into an `AgentCallToAction` — the agent's unit of work:

```python
# Each org position → one AgentCallToAction form
# Each metric the position tracks → one ActionFormField the agent must fill
# Each gate the position needs → one gate_id checkpoint on the action

AgentCallToAction(
    action_id   = "action_a3f2b1c0",
    agent_id    = "agent_field_technician",
    action_name = "field_technician_status_report",
    fields = [
        ActionFormField(field_id="jobs_completed_today",  ...),
        ActionFormField(field_id="materials_cost_actual", ...),
        ActionFormField(field_id="customer_satisfaction", ...),
    ],
    gate_ids = ["safety_gate", "quality_gate", "budget_gate"],
)
```

**This is the automation pipeline.** The `AgentCallToAction` describes exactly what the agent needs to know and what gates it must clear for each role in the business. The pipeline is not hardcoded — it is generated from the filled form.

When the agent runs, sensors observe the business state and fill the `ActionFormField` values:
- `SensorType.EVENT_STREAM` — job completion events from the dispatch system
- `SensorType.API_RESPONSE` — materials costs from the supplier API
- `SensorType.LLM_INFERENCE` — inferred satisfaction from customer reply text (gated)
- `SensorType.USER_INPUT` — technician taps a confirmation in the mobile app

Each `LLM_INFERENCE` field has `needs_gating = True` and waits for the confidence gate to pass before it is accepted as a constraint. `VERIFIED` and `HIGH_CONFIDENCE` fields pass immediately.



Two layers enforce that no LLM output escapes the customer's domain:

**Layer A — Input lock: `PromptAmplifier.amplify_for_niche()`**  
`src/prompt_amplifier.py`

Before any LLM call is made for a customer agent, the raw prompt is passed through `amplify_for_niche(raw_prompt, niche_description)`. This prepends the niche context to the prompt so the LLM is always answering in the context of the specific business:

```
niche_description = "Plumbing and drain cleaning service business in Austin TX"
raw_prompt        = "Write a follow-up message to a customer"

→ amplified: "Plumbing and drain cleaning service business in Austin TX — Write a follow-up
              message to a customer [+ magnified domain-specific components]"
```

The LLM sees the niche context first. It will write a plumbing follow-up, not a generic one, and certainly not an aerospace one.

**Layer B — Output gate: `SafeLLMWrapper._check_relevance()`**  
`src/safe_llm_wrapper.py`

After the LLM responds, `_check_relevance()` computes Jaccard similarity between the prompt tokens and the response tokens. If similarity falls below 0.05 (5% overlap) on a response longer than 50 characters, the gate fails and the response is blocked:

```python
# Rule 1: Completely off-topic — very low overlap on a substantial response
if jaccard < 0.05 and len(response) > 50:
    return False  # gate fails → SafetyGate blocks output → fallback response used
```

A plumbing prompt that gets an aerospace response will have near-zero Jaccard overlap and will be caught here. The fallback response is returned instead and the failure is logged.

### Step 4 — Gate Lifecycle: Generated → Active → Retired

`src/gate_synthesis/models.py` — `GateState`, `Gate`  
`src/gate_synthesis/gate_lifecycle_manager.py` — `GateLifecycleManager`

Every gate generated by `InferenceDomainGateEngine` (or `DomainGateGenerator`) enters the lifecycle at `PROPOSED` and must be explicitly activated before it enforces:

```
PROPOSED   Gate exists as an artifact. Not yet enforcing.
    │      GateLifecycleManager.add_gate(gate) registers it.
    ▼
ACTIVE     GateLifecycleManager.activate_gate(gate_id) promotes it.
    │      Gate is now live — its trigger_condition is evaluated on every
    │      relevant task. enforcement_effect (block / require_evidence /
    │      escalate) applies when the condition is met.
    ▼
SATISFIED  gate.check_retirement_conditions() returns True.
    │      All RetirementCondition thresholds have been met
    │      (e.g., N consecutive passes, or metric stabilised above threshold).
    ▼
RETIRED    GateLifecycleManager retires the gate.
           Logged to retirement_log with timestamp and reason.
           GovernanceKernel removes it from the active gate set.

     (also: EXPIRED — gate reached its expires_at datetime without being satisfied)
```

For customer-specific business context gates, the operator reviews the generated gate set in the HITL terminal before any gate moves from `PROPOSED` to `ACTIVE`. `activate_all_proposed_gates()` is available for bulk activation after review.

### Step 5 — Automation Activation: Manual → Supervised → Automated

`src/hitl_graduation_engine.py` — `HITLGraduationEngine`, `HITLRegistry`

Once a gate is `ACTIVE` and its act-function begins routing tasks to capabilities, those capabilities start under manual oversight. `HITLRegistry` tracks each automation as an `HITLItem` with a `current_mode`:

```
manual       Every execution requires explicit operator approval.
             Operator sees: "Ready to run 'send_job_estimate' — approve? [Y/N]"

supervised   Executions run automatically but operator is notified of every result.
             Operator sees: "Ran 'send_job_estimate' — result: sent to customer. [OK/Rollback]"

automated    Executions run silently. Operator notified only on failure or anomaly.
```

`HITLGraduationEngine.evaluate_item()` computes a `graduation_score` from:
- `success_rate` — fraction of executions with good outcomes
- `risk_score` — potential blast radius if it goes wrong
- `impact_score` — positive value delivered

When `graduation_score >= SUPERVISED_THRESHOLD (0.60)`, `HITLGraduationEngine.recommend_mode()` recommends graduating from `manual` to `supervised`. The operator approves via `HITLRegistry.graduate_item(item_id, new_mode)`. Graduation to `automated` requires a higher threshold and always requires explicit operator approval — it is never automatic.

`HITLRegistry.rollback_item()` reverts to the previous mode immediately if something goes wrong.

### Step 6 — Pattern Solidification: Shadow Learning → Workflow Patterns

`src/org_compiler/shadow_learning.py` — `TelemetryCollector`, `ShadowLearner`  
`src/trading_shadow_learner.py` — `PatternMemoryStore`, `WeeklyWinningPattern`  
`src/rosetta/rosetta_models.py` — `WorkflowPattern`, `ImprovementProposal`  
`src/rosetta/rosetta_manager.py` — `RosettaManager.update_state()`

Shadow agents observe without acting. As the operator approves automations and tasks execute, the shadow layer records what happened:

```
TelemetryCollector.record_task_assignment(role, task, timestamp)
TelemetryCollector.record_approval(role, approval_type, granted, timestamp)
TelemetryCollector.record_handoff(handoff_event)
```

After each observation window, `ShadowLearner.observe_role()` analyses the recorded patterns and produces a `TemplateProposalArtifact` — a proposed automation template based on what the operator repeatedly did the same way. This maps to an `ImprovementProposal` in `RosettaAgentState`.

The operator reviews proposals in the HITL panel. If approved, the proposal is promoted to a `WorkflowPattern` and written back via `RosettaManager.update_state()`:

```python
RosettaManager.update_state(agent_id, {
    "workflow_patterns": [
        {
            "pattern_id":           "send-estimate-after-site-visit",
            "name":                 "Send estimate within 2h of site visit",
            "steps":                ["complete_site_visit", "generate_estimate", "send_estimate"],
            "success_rate":         0.91,
            "avg_duration_seconds": 420,
            "usage_count":          17,
        }
    ]
})
```

The `WorkflowPattern` is now part of the agent's permanent locked context. The Librarian can reference it when routing similar tasks — preferring this known-good pattern over an untested alternative. The `success_rate` and `usage_count` feed directly into `FeedbackIntegrator` weights for future routing decisions.

## HITL Interaction with Solution Path Selection

When a `SolutionPath` has `requires_hitl=True`, or when a gate returns `"hitl"` status, the `HITL Graduation Engine` intercepts execution:

```
TaskRouter returns HITL result
          │
          ▼
HITLGraduationEngine.request_approval(routing_result)
          │
          ├── Presents all SolutionPath alternatives to operator:
          │     "I found 3 ways to do this:
          │      1. invoice_processing_pipeline  (score 0.94, deterministic, low cost)
          │      2. niche_business_generator     (score 0.71, stochastic, medium cost)
          │      3. sales_automation             (score 0.58, deterministic, low cost)
          │      Which would you like to use? [1/2/3/auto]"
          │
          ├── Operator selects path (or approves auto-select)
          │
          └── Selected path proceeds to execution
                    │

                              │
                              └── future tasks with same pattern
                                  → selected path ranked higher
```

Over time, operator selections train the routing system. Capabilities that operators consistently prefer over higher-scored alternatives are boosted in the `FeedbackIntegrator` weight table.

```

      │
      ▼
SolutionPathRegistry.record_outcome(task_id, path_id, success=True)
      │
      ▼
FeedbackIntegrator.ingest(capability_id, success=True, latency_ms=120)
      │
      ▼
FeedbackIntegrator._update_weight_table(capability_id)
  weight = (successes + 1) / (total_attempts + 2)  # Laplace smoothing
      │
      ▼
TaskRouter._rank_paths() queries FeedbackIntegrator.get_path_score(capability_id)
      │
      └── combined_score = librarian_score × historical_weight
```

This creates a fully closed learning loop:
- New capabilities start with neutral weight (1.0).
- Successes increase weight toward 1.0.
- Failures decrease weight toward 0.0.
- The Librarian's semantic ranking is preserved but modulated by real-world performance.

## `librarian_bot` TypeScript Bridge

The `librarian_bot` TypeScript service provides a higher-quality semantic search than pure Python keyword matching. It is called via the existing `LibrarianAdapter`:

```python
class LibrarianAdapter:

        top_n: int = 5,
    ) -> List[CapabilityMatch]:
        """

        Falls back to keyword matching if the service is unavailable.
        """
        ...
```

The `librarian_bot` is started as part of `setup_and_start.sh`. If it is unavailable (e.g., Node.js not installed), `SystemLibrarian.find_capabilities()` falls back to keyword + domain-tag matching transparently.
